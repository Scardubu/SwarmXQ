# 5. PRODUCTION CERTIFICATION TIERS

Use persisted, typed, observable lifecycle states:

1. `RENDER_FAILED`
2. `TECHNICALLY_VALID`
3. `CREATIVE_REVIEW_REQUIRED`
4. `PRODUCTION_PACK_VALID`

6. `PUBLISHING`
7. `PUBLISHED_VERIFIED`
8. `PUBLISH_FAILED`
9. `BLOCKED`
10. `NEEDS_REVISION`

Definitions:

- `TECHNICALLY_VALID` means valid streams and basic media properties only.
- `CREATIVE_REVIEW_REQUIRED` means technical checks pass but creative, audio, accessibility, rights, continuity, or platform requirements remain.
- `PRODUCTION_PACK_VALID` means script, assets, voice, captions, manifests, QC, and platform package are complete; approval may remain.
- `READY_TO_POST` means all mandatory technical, creative, accessibility, rights, provenance, and platform gates pass and the exact package is approved.
- `PUBLISHED_VERIFIED` requires verified remote processing, final visibility, metadata, disclosure, and canonical remote ID or URL.

No field named `complete` may ambiguously mean “some output exists.”

# 13. VOICE AND AUDIO PRODUCTION

## 13.1 `VoiceProvider`

Replace renderer-owned direct TTS execution with:

```ts
interface VoiceProvider {
  id: string;
  probe(): Promise<VoiceCapability>;
  listVoices(): Promise<VoiceDescriptor[]>;
  synthesize(request: VoiceSynthesisRequest): Promise<VoiceArtifact>;
  cancel?(operationId: string): Promise<void>;
}
```

Providers:

- `PiperVoiceProvider`
- `EspeakVoiceProvider`
- optional `KokoroVoiceProvider`
- explicit technical-test silence provider, never a production voice substitute

## 13.2 Selection policy

Benchmark on actual profiles:

- install size;
- model size;
- cold start;
- warm synthesis latency;
- peak memory;
- real-time factor;
- repeated-request stability;
- sample rate;
- intelligibility;
- pronunciation;
- expressiveness;
- available voices and languages;
- licensing;
- process lifecycle.

Prefer a persistent local server/process when repeated CLI invocation reloads the voice model unnecessarily.

No provider becomes the default solely because it sounds better on a different machine.

## 13.3 Voice identity

A selected voice must be a real provider/voice ID, not merely a rate alias such as “calm” or “energetic.”

Separate:

- semantic voice role;
- provider;
- voice ID;
- locale;
- speaking rate;
- pitch/prosody controls;
- pronunciation dictionary.

Persist all values.

## 13.4 Script normalization

Before synthesis:

- remove unmatched decorative quotes;
- normalize punctuation;
- expand unsafe abbreviations when required;
- preserve intentional acronyms;
- split sentences semantically;
- validate language and locale;
- apply pronunciation dictionary;
- reject hidden reasoning tags and prompt markup;
- retain canonical source text.

## 13.5 Audio post-production

Use deterministic FFmpeg services for:

- resampling;
- channel layout;
- high-pass or low-pass filtering when justified;
- EQ;
- compression;
- dialogue ducking;
- fades;
- silence placement;
- limiting;
- loudness normalization;
- final muxing.

Prefer measured/two-pass loudness normalization when practical.

Define platform-aware audio profiles with:

- target integrated loudness;
- true-peak ceiling;
- loudness range guidance;
- sample rate;
- channel policy;
- minimum dialogue intelligibility.

## 13.6 Transcript and alignment

Optionally use a lightweight local ASR adapter such as whisper.cpp for:

- transcript verification;
- subtitle timing;
- missing-word detection;
- pronunciation QA.

Keep it optional on constrained hardware. ASR disagreement must produce a review signal, not silently rewrite the script.

## 13.7 Music and SFX

- use rights-safe local libraries or explicitly licensed imports;
- preserve attribution;
- enforce dialogue/music balance;
- use a sound-event timeline;
- support intentional silence;
- prevent repetitive sound signatures across unrelated projects;
- block unresolved music rights from final certification.
